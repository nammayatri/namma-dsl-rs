# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`namma-dsl-rs` is a single-binary code generator (CLI). It reads YAML "Storage" specs and emits a full Rust + Diesel persistence layer: domain types, `diesel::table!` schema, `Queryable`/`Insertable` model structs, typed query functions, and SQL migrations. It is the Rust port of the Haskell `namma-dsl` storage generator (see Haskell‚ÜíRust type mapping in README.md).

## Build / run / test

```bash
cargo build                            # build CLI
cargo install --path .                 # install as `namma-dsl-rs` to ~/.cargo/bin
cargo fmt                              # the binary auto-runs this on its target repo unless --skip-fmt
```

Run the generator against the in-repo fixture (this is the de facto end-to-end test ‚Äî there are no `cargo test` unit tests yet):

```bash
cargo run -- --path test-fixture/spec --skip-fmt
# Outputs land under test-fixture/src-read-only/ (gitignored).
# Diff against committed reference output if you want to spot regressions.
```

When invoked inside a target service repo (no `--path`), the binary walks for any directory named `spec/` containing a `dsl-config.toml` and processes every `Storage/*.yaml` whose git hash differs from `HEAD` (or all of them with `--all`). It requires a git working tree ‚Äî `find_git_root` (`src/utils.rs`) shells out to `git rev-parse`.

## Architecture

The pipeline has three distinct stages, mirrored by the module layout:

```
YAML spec  --(parser)-->  TableDef IR  --(generator)-->  Rust/SQL strings  --(main)-->  files on disk
```

1. **`src/parser/storage.rs`** ‚Äî Parses one `Storage/*.yaml` into a `Vec<TableDef>`. A single YAML file can declare multiple top-level models (e.g. `Seat.yaml` declares both `Seat` and `SeatLayout`); the top-level `imports:` key is shared across all models in that file. Where-clauses are recursively parsed into a `WhereClause` tree (`Leaf` / `And` / `Or` / `In`). The `extra_default_fields` from `dsl-config.toml` (typically `createdAt`/`updatedAt`) are appended here, not in the generators.
2. **`src/types/storage.rs`** ‚Äî The IR. `TableDef` is what every generator consumes; once a YAML is parsed, generators never touch `serde_yaml` again.
3. **`src/generator/{domain_type, diesel_schema, diesel_model, queries, sql}.rs`** ‚Äî Five independent string-emitters, one per `GeneratorKind`. They share helpers via `src/utils.rs`. Each is gated on `[generate].generators` in `dsl-config.toml`, so a service can opt out of any of them.
4. **`src/main.rs`** ‚Äî CLI + filesystem orchestration. Walks for `spec/` dirs, calls the parser, fans out to enabled generators, writes files to paths from `[output]` resolved against the git root, then runs `cargo fmt` on the target repo.

### Three parallel type representations

This is the central concept ‚Äî every field has three forms, and the generators translate between them. Helpers live in `src/utils.rs`:

| Layer | Examples | Used in |
|---|---|---|
| **Domain type** (Rust author-facing) | `Id<Seat>`, `Option<SeatType>` (enum), `chrono::DateTime<Utc>`, `Vec<Foo>` | `domain_type.rs`, query function signatures |
| **DB model type** (`*Model` / `New*` structs) | `String`, `Option<String>`, `chrono::NaiveDateTime`, `serde_json::Value` | `diesel_model.rs` (`domain_type_to_db_type`) |
| **Diesel column type** (`table!` macro) | `Varchar`, `Nullable<Text>`, `Timestamptz`, `Jsonb` | `diesel_schema.rs` (`rust_type_to_diesel_type`) |

`generator/queries.rs` is where conversions get emitted (`From<Model> for Domain` and `From<&Domain> for NewModel`) ‚Äî see `model_field_to_domain` / `domain_field_to_model` / `entity_field_to_db_value`. The rules: `Id<X>` ‚Üî `String` (via `Id::new` / `.inner().to_string()`), enums ‚Üî `String` (via `Display` / `FromStr`, both auto-generated by `domain_type.rs`), `DateTime<Utc>` ‚Üî `NaiveDateTime` (via `.naive_utc()` / `.and_utc()`).

When adding a new domain type, all three mapping helpers in `utils.rs` likely need a branch, plus the conversion helpers in `queries.rs`.

### kvFunction ‚Üí query generator dispatch

`KvFunction` (in `types/storage.rs`) is the shared vocabulary between YAML and the generator. The match in `parser/storage.rs::parse_queries` accepts both `*WithKV` (Haskell DSL legacy) and bare names (`findOne`, `findAll`, etc.) and collapses them onto the same enum. `generator/queries.rs::generate_spec_query` then dispatches on that enum.

`FindOnePartial` and `FindAllPartial` arms are currently empty ‚Äî they parse but emit nothing. If a spec uses them, queries will silently be missing.

In addition to user-declared queries, every table gets four auto-generated query helpers: `create`, `create_many`, `find_by_primary_key`, `update_by_primary_key` (the last two are skipped if there's no PK / no non-PK fields).

### SQL generator is incremental

`generator/sql.rs` reads any existing migration file at the target path and only appends `ALTER TABLE ‚Ä¶ ADD COLUMN` for columns it doesn't find (parsed by string-scanning lines containing `ADD COLUMN`). It never drops or alters existing columns. Fresh files emit `CREATE TABLE ... ();` followed by per-column `ALTER TABLE`s ‚Äî this matches the Haskell DSL's migration style and is intentional, not a bug.

### Change detection

`utils.rs::get_file_state` compares `git ls-tree HEAD <path>` to `git hash-object <path>` to decide whether a YAML changed since the last commit. `--all` bypasses this. Files not tracked by git read as `New`. This means uncommitted edits to a YAML *do* trigger regeneration on the next run (good), but renaming a YAML on disk without staging may produce surprising results.

## Conventions worth knowing

- **Field naming**: YAML uses camelCase (`seatLayoutId`), Rust uses snake_case. The parser stores both (`field_name` / `field_name_snake`); SQL column names are always snake_case (via `heck`).
- **Output paths in `dsl-config.toml`** are resolved relative to the git root, with `$GIT_ROOT` token expansion supported (`config.rs::resolve_path`).
- **No tests yet.** The `test-fixture/` directory is the only validation surface ‚Äî its `src-read-only/` is gitignored, so `git status` after a run shows nothing if the fixture compiles.
- **Don't edit generated files.** Every emitted file starts with `// AUTO-GENERATED by namma-dsl-rs. Do not edit.`. The fix is in the generator, not the output.
- **`src/types/mod.rs`** currently only re-exports `storage`. Keep new IR types in `types/storage.rs` unless adding a new spec category (e.g. an "API" generator).
